## 五、技术选型评估回顾

### 5.1 Embedding 模型选型

| 指标 | nomic-embed-text | bge-m3 | 胜者 |
|------|-----------------|--------|------|
| 向量维度 | 768 | 1024 | bge-m3 |
| Recall@5 | 44.44% | 100.00% | bge-m3 |
| MRR | 29.81% | 97.22% | bge-m3 |
| 查询耗时 | 0.091s | 0.375s | nomic |
| 中文支持 | 差（英文模型） | 优秀（多语言） | bge-m3 |

**结论：** bge-m3 在中文检索场景下碾压 nomic-embed-text，虽然慢4倍但完全够用。

### 5.2 分块策略选型

| chunk_size | overlap | chunks数 | Recall@5 | 选择 |
|-----------|---------|---------|----------|------|
| 200 | 20 | 818 | 100% | 太碎 |
| 500 | 50 | 318 | 100% | **选定** |
| 800 | 80 | 191 | 100% | 可接受 |
| 1000 | 100 | 157 | 100% | 太大 |

**结论：** 由于 bge-m3 太强，所有参数都达到 100%。选择 500/50 是工程最优平衡。

### 5.3 向量数据库选型

选择 Milvus 的理由：
1. Docker 已部署，零额外成本
2. 企业级首选，展示生产级能力
3. HNSW 索引适合中小规模数据
4. PyMilvus 3.0 API 简洁

### 5.4 Agent 架构选型

选择 Supervisor 模式的理由：
1. 面试高频考点
2. 职责清晰，易扩展
3. 三种任务类型适合路由模式
4. 代码量适中，适合简历展示
