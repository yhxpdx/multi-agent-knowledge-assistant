## 一、RAG (检索增强生成)

### Q1: 请介绍你的 RAG 管道的完整流程

**参考回答**:
我的 RAG 管道分为索引和检索两个阶段：

索引阶段：
1. 文档上传后，根据文件类型选择解析器（PDF 用 PyPDF，Markdown 直接读取）
2. 使用 RecursiveCharacterTextSplitter 进行文本分块，chunk_size=500，overlap=50
3. 调用 bge-m3 模型生成 1024 维的 embedding 向量
4. 将 chunks 和向量存入 Milvus，使用 HNSW 索引和 COSINE 相似度

检索阶段：
1. 用户查询经过 bge-m3 生成向量
2. 在 Milvus 中执行 Top-K 向量搜索（默认 K=5）
3. 将检索结果格式化为带来源引用的上下文
4. 将上下文和用户问题一起输入 LLM 生成回答

### Q2: 为什么选择 chunk_size=500？这个参数是怎么确定的？

**参考回答**:
这不是拍脑袋决定的，而是通过实验评估得出的。我对比了 7 种参数组合：
- chunk_size: 200/500/800/1000
- overlap: 20/50/80/100/150

用 10 个测试查询评估 Recall@5 和 MRR。结果显示所有参数组合都达到 100% Recall@5（因为 bge-m3 太强了），但从工程角度：
- 200 字太碎：产生 818 个 chunks，存储和检索开销大
- 1000 字太大：可能包含无关信息，稀释关键内容
- 500 字是平衡点：318 个 chunks，424 字平均长度，足够包含完整语义

这也是 LangChain 官方推荐的默认起始值，有社区验证。

### Q3: 你的分块策略是什么？为什么选 RecursiveCharacterTextSplitter？

**参考回答**:
我选择 RecursiveCharacterTextSplitter，原因是：
1. 它按分隔符层级递归分割（\n\n → \n → 句号 → 空格），尽量保持语义完整性
2. 支持自定义分隔符，我针对中文添加了"。""！""？"等分隔符
3. 是 LangChain 推荐的默认分块器，社区使用广泛

对比其他方案：
- 固定大小分块：简单但可能切断句子
- 语义分块：效果最好但计算开销大，需要调用 Embedding 模型
- 文档结构分块：适合结构化文档，但对纯文本不适用

### Q4: 如何解决 RAG 中的幻觉问题？

**参考回答**:
我在 prompt 中明确要求 LLM"只基于提供的文档内容回答，不要编造信息"，并且在回答中引用来源。此外：
1. 检索结果包含来源文件名和 chunk 编号，便于验证
2. 如果检索结果不相关，Agent 会明确告知"未找到相关信息"
3. 设置了相似度阈值，低于阈值的结果不纳入上下文

### Q5: 向量搜索和混合搜索有什么区别？你为什么选择纯向量搜索？

**参考回答**:
- 纯向量搜索：基于 embedding 语义相似度，擅长理解意图
- 混合搜索：结合向量搜索和关键词搜索（BM25），兼顾语义和精确匹配

我选择纯向量搜索的原因：
1. bge-m3 在我的测试集上 Recall@5 达到 100%，已经足够好
2. 混合搜索需要额外配置（BM25 索引、权重调参），增加复杂度
3. 我的知识库是技术文档，语义检索比关键词匹配更合适

但如果知识库包含大量专有名词（如产品型号、法律条款），混合搜索会更好。

## 二、Agent 架构

### Q6: 为什么选择 Supervisor 模式？和其他模式对比如何？

**参考回答**:
我对比了三种多智能体架构：

| 模式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| Supervisor | 职责清晰，易扩展 | Supervisor 可能成瓶颈 | 任务类型明确，需专业分工 |
| Swarm | 灵活，无中心瓶颈 | 控制流复杂，调试难 | Agent 频繁交互 |
| Hierarchical | 适合大规模任务 | 层数多延迟高 | 企业级复杂工作流 |

选择 Supervisor 的原因：
1. 面试高频考点，最常见的多智能体架构
2. 我的场景是三种明确的任务类型（文档问答、联网搜索、代码），适合路由
3. 代码量适中，适合简历项目展示
4. 新增子 Agent 只需添加节点和边，扩展性好

### Q7: Supervisor 如何做路由决策？

**参考回答**:
Supervisor 使用 LLM 进行意图判断：
1. 将用户查询和可用 Agent 列表发送给 LLM
2. LLM 分析查询内容，返回 JSON 格式的路由决策：{"agent": "document_qa"}
3. 解析 JSON，验证 agent 名称合法性
4. 使用 LangGraph 的条件边路由到对应子 Agent

System Prompt 设计要点：
- 明确列出每个 Agent 的职责和适用场景
- 要求只返回 JSON，不要返回其他内容
- 设置 temperature=0 确保路由一致性

### Q8: 如果 Supervisor 路由错了怎么办？

**参考回答**:
1. 首先，我设置了默认路由（document_qa），如果 JSON 解析失败就走默认
2. 子 Agent 内部也有容错：如果文档检索没结果，会明确告知用户
3. 用户可以重新表述问题，系统会重新路由
4. 未来可以增加路由置信度阈值，低置信度时询问用户

### Q9: LangGraph 的 StateGraph 是什么？你在项目中怎么用的？

**参考回答**:
StateGraph 是 LangGraph 的核心概念，它定义了：
- State：贯穿整个执行过程的共享状态（TypedDict）
- Node：处理函数，执行具体逻辑
- Edge：节点间的连接，支持条件路由

我的项目中的状态定义：
```python
class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]  # 消息历史
    next_agent: str  # 下一个 Agent
    context: str  # RAG 上下文
```

图结构：
- entry → supervisor → (条件路由) → document_qa / web_search / code_assistant → END

## 三、向量数据库

### Q10: 为什么选择 Milvus？和 ChromaDB、FAISS 对比如何？

**参考回答**:
我从五个维度对比：

| 维度 | Milvus | ChromaDB | FAISS |
|------|--------|----------|-------|
| 架构 | 分布式 | 轻量级 | 纯内存 |
| 数据规模 | 百万级 | 十万级 | 百万级 |
| 部署 | Docker/K8s | 嵌入式 | 库 |
| 索引类型 | HNSW/IVF等 | HNSW | IVF/HNSW |
| 生产就绪 | 是 | 原型 | 需封装 |

选择 Milvus 的原因：
1. 已有 Docker 部署，零额外成本
2. 项目定位是展示生产级能力，Milvus 是企业首选
3. 支持丰富的索引类型，可以做性能对比实验
4. Python SDK 完善，与 LangChain 集成好

### Q11: HNSW 索引是什么？为什么选它？

**参考回答**:
HNSW (Hierarchical Navigable Small World) 是一种基于图的近似最近邻搜索算法：
- 构建多层图结构，每层是一个 NSW 图
- 搜索时从顶层开始，逐层向下导航
- 时间复杂度 O(log n)，空间复杂度 O(n)

选择 HNSW 的原因：
1. 我的数据规模（几百到几千条）HNSW 最合适
2. 不需要训练（不像 IVF 需要 nlist 参数）
3. 查询速度快，P99 延迟通常 <10ms
4. 参数简单：M=16, efConstruction=200

对比 IVF_FLAT：
- IVF_FLAT 需要设置 nlist，小数据集不好调
- IVF_FLAT 需要先训练再插入，HNSW 不需要
- HNSW 内存占用更高，但我的数据规模小，不是问题

### Q12: Milvus 的 COSINE 相似度和 L2、IP 有什么区别？

**参考回答**:
- COSINE：计算向量夹角的余弦值，范围 [-1, 1]，适合文本语义相似度
- L2 (欧氏距离)：计算向量间的直线距离，范围 [0, +∞)，适合图像特征
- IP (内积)：计算向量点积，范围 [-∞, +∞)，适合归一化后的向量

我选择 COSINE 的原因：
1. Embedding 模型通常输出归一化向量，COSINE 和 IP 效果相同
2. COSINE 的结果范围固定 [-1, 1]，便于设置阈值
3. 文本语义相似度天然适合用角度衡量

## 四、Embedding 模型

### Q13: 你做了什么 Embedding 模型评估？结果如何？

**参考回答**:
我用实际知识库数据（25 篇文档）构建了 18 个测试查询，对比了两个模型：

| 指标 | nomic-embed-text | bge-m3 |
|------|-----------------|--------|
| 向量维度 | 768 | 1024 |
| Recall@5 | 44.44% | 100.00% |
| MRR | 29.81% | 97.22% |
| 查询耗时 | 0.091s | 0.375s |

关键发现：
- nomic-embed-text 英文检索完全失效（0% Recall），因为它是英文为主的模型
- bge-m3 中英文都是 100%，因为它专门针对多语言优化
- bge-m3 虽然慢 4 倍，但 0.375s 对 RAG 场景完全够用

### Q14: 为什么用 Ollama 本地模型而不是 API？

**参考回答**:
1. 火山引擎 coding plan 不支持 Embedding API（返回 404）
2. 本地模型无网络延迟，无 API 调用成本
3. bge-m3 评估结果优秀，不需要依赖外部服务
4. Ollama 已有 Docker 部署，拉取模型即可使用

## 五、工具调用 (Function Calling)

### Q15: Function Calling 的工作原理是什么？

**参考回答**:
1. 工具定义：开发者定义工具的名称、描述、参数 schema（JSON Schema）
2. 工具选择：LLM 根据用户请求和工具描述，决定是否调用工具
3. 参数生成：LLM 生成符合 schema 的结构化参数
4. 工具执行：系统实际调用工具函数
5. 结果整合：将工具返回值注入对话上下文，LLM 生成最终回答

### Q16: 你的代码执行工具如何保证安全？

**参考回答**:
我实现了多层安全措施：
1. 模块黑名单：禁止 import os, sys, subprocess, shutil 等危险模块
2. 函数黑名单：禁止 open(), exec(), eval(), __import__() 等
3. 受限的 builtins：从 __builtins__ 中移除危险函数
4. 输出限制：stdout 最多 2000 字符，防止内存溢出
5. 错误捕获：所有异常被捕获并返回友好错误信息

### Q17: 工具描述怎么写才能让 LLM 正确选择？

**参考回答**:
工具描述要精确、具体、有区分度：

差：「搜索文档」
好：「从知识库中搜索与查询相关的文档片段，返回 Top-K 结果及其相似度分数。当用户询问已上传文档相关问题时使用。」

关键原则：
1. 说明工具的功能和返回内容
2. 说明适用场景（什么时候用这个工具）
3. 参数描述要具体（类型、范围、默认值）
4. 避免描述模糊或重叠

## 六、工程实践

### Q18: 你的 FastAPI 后端如何实现流式响应？

**参考回答**:
使用 SSE (Server-Sent Events) 实现：
1. 客户端发送 POST /api/chat，stream=true
2. 服务端返回 Content-Type: text/event-stream
3. 使用 StreamingResponse 逐 token 发送
4. 每个 token 格式：data: {"content": "...", "node": "..."}\n\n
5. 完成后发送：data: {"done": true, "session_id": "..."}\n\n

前端使用 requests 库的 iter_lines() 逐行接收。

### Q19: Redis 在你的项目中怎么用的？

**参考回答**:
Redis 用于对话记忆管理：
- 数据结构：List 存储消息序列，Hash 存储会话元数据
- Key 设计：chat:{session_id}:messages, chat:{session_id}:meta
- TTL：7 天自动过期，每次访问刷新
- 截断策略：保留最近 20 条消息（10 轮对话）

为什么用 List 而不是 String？
- List 支持 RPUSH/LRANGE，天然适合消息队列
- 可以用 LTRIM 截断，不需要每次序列化整个历史

### Q20: 你的项目有什么可以改进的地方？

**参考回答**:
1. 检索优化：实现混合搜索（向量 + 关键词），提升专有名词检索效果
2. 重排序：增加 Reranker 模型，对检索结果二次排序
3. 评估体系：建立自动化评估管道，持续监控 RAG 质量
4. 缓存策略：缓存高频查询的 embedding 和检索结果
5. 多轮对话：优化上下文窗口管理，支持更长的对话历史
6. 用户反馈：增加点赞/点踩功能，收集用户反馈优化系统
